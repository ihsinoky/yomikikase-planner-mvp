# 日程確定モック Review メモ（MVP 観点）

## 参照情報
- SystemDesign.md 7.1「日程確定 UI モック」の要件（学年フィルタ→日付決定→参加者確定の最小フロー）。
- 現行モック: `mockups/schedule-confirmation.html`（モバイル前提レイアウト、CDN Alpine.js、フェッチなしの console.log）。

## MVP で残すべき要素
- **学年フィルタ**: 指定学年のみを選出する体験は SystemDesign の最小フローに含まれるため維持。
- **候補日選択とハイライト**: 選択状態が明示されないと参加者確定の対象日が分からないため必要。視覚ハイライトはモックでも残す。
- **参加者チェックボックス**: MVP の「日付ごとに参加者を確定」操作の本丸。ID 配列で送信する現在の挙動を維持。
- **送信結果の即時フィードバック**: API がまだ無いので console.log + 画面内ログ表示はステークホルダー説明用に必須（副作用なし）。

## MVP で簡素化してよい／後回しでよい要素
- **装飾強度**（影やボタン色の濃さ）: 操作可能性を損なわない範囲なら簡素化可。現状も最小限の CSS のため追加削減は不要だが、要望があれば影を薄くする程度で対応可能。
- **PC レイアウトの最適化**: MVP はスマホ前提。640px 以上のグリッドは「モバイルでの確認を阻害しない範囲」で残しているが、不要なら単一カラム固定でも可。
- **ダミーデータの網羅性**: 現在は各学年 1〜2 名のサンプルでフィルタ動作を示せるため十分。実人数に近づけるのは本実装以降でよい。

## 結論
- レビューコメントが「装飾過多では？」という観点なら、MVP 要件は機能体験の確認なので**現状の軽量スタイルを維持しても許容範囲**。さらに削る場合は影の弱体化程度で済む。
- 機能フロー（フィルタ→日付選択→参加者確定）は SystemDesign 由来の必須要素のため、MVP でも削らず保持するのが妥当。
